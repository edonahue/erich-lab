# Erich's Lab Contract

This document defines the durable expectations for experiments published through `lab.erichdonahue.com`. It is intentionally proportional: a one-file toy should remain easy to publish, while a data-backed or networked lab should document the additional responsibilities it introduces.

## Experiment levels

### Level 1: Tiny artifact

Examples: a self-contained HTML toy, visual sketch, calculator, or static reference.

Required:

- Public Markdown entry under `src/content/experiments/`
- Stable slug and public launch route
- Source link
- Mobile usability check
- Dark and light theme behavior
- Clear status and update date
- No secrets or private operational details

### Level 2: Interactive lab

Examples: quizzes, games, browser tools, and experiments with local state.

Adds:

- `experiment.json` packaging manifest
- Keyboard-accessible interaction and visible focus
- Playwright smoke coverage
- Local-storage key and reset behavior
- Explicit offline status
- Graceful handling when optional features are unavailable
- Inner application marked `noindex`; public wrapper remains indexable

### Level 3: Data or backend lab

Examples: live APIs, scheduled data products, home-hosted services, and experiments using third-party datasets.

Adds:

- Experiment brief
- Data-source and rights review
- Public/private boundary
- Failure and degradation behavior
- Architecture decision records for consequential choices
- Operational notes in a private repository when they expose infrastructure
- Provenance, versioning, recovery, and observability expectations

## URL and content structure

Each published experiment uses one slug:

```text
src/content/experiments/<slug>.md
public/experiments/<slug>/experiment.json
public/experiments/<slug>/<entry file>
/projects/<slug>/
/experiments/<slug>/
```

The Markdown file owns editorial metadata and public project notes. The JSON manifest owns technical packaging: entry file, wrapper links, offline inputs, compatibility paths, and smoke-test hints. Do not duplicate architecture prose or private operations in the manifest.

## Public wrapper requirements

Wrappers are generated by `scripts/build-experiments.mjs` and must provide:

- Link back to the lab
- Link to project notes
- Link to the main website
- Declared downloads
- Universal theme toggle
- iframe title
- Canonical wrapper URL

The wrapper is the indexable launch page. The inner application and downloadable HTML are implementation artifacts and must be `noindex`.

## Theme contract

```text
Storage key: erich-lab-theme
Root attribute: data-theme="dark" | data-theme="light"
Default: dark
```

Every experiment must remain legible in both modes. A distinct visual identity is welcome, but ignoring the shared preference is not.

## Accessibility baseline

Interactive experiments must provide:

- Keyboard access to all controls
- Visible focus states
- Semantic labels for inputs
- Correct tab, dialog, disclosure, and status semantics where applicable
- Reduced-motion behavior when animation is not essential
- Mobile layouts that do not hide essential actions

The smoke suite is not a complete accessibility audit. Manual keyboard and screen-reader spot checks remain part of release review.

## Local state

Local-storage keys must be experiment-specific and versioned when data shape changes. A reset control is required when stored state materially affects the experience. Migrations should preserve compatible progress when practical and explicitly reset incompatible state.

## Offline support

Use these labels consistently:

- **Full:** the declared offline edition preserves the experiment's core experience.
- **Limited:** the edition intentionally omits live services, large datasets, or other documented functionality.
- **Unavailable:** no download is declared.

Offline artifacts are generated, not hand-edited. External documentation links may require connectivity even when the experiment itself is self-contained.

## Indexing and discovery

Index:

- Lab homepage
- Project-note pages
- Public experiment wrappers

Do not index:

- Inner app entry files
- Download artifacts
- Compatibility redirects
- Private operational documentation

Published wrappers and project pages belong in the sitemap. Archived and graduated experiments may retain stable lab pages when that history remains useful.

## Public and private boundary

Public pages may explain roles, technologies, algorithms, tradeoffs, aggregate findings, and reproducible design decisions. Keep private:

- Credentials and tokens
- Hostnames and addresses
- Exact port maps and physical topology
- Personal collection membership unless intentionally published
- Backup destinations and restore commands
- Tunnel configuration
- Incident details that increase attack surface

## Data and media rights

Before publishing data, images, audio, or generated derivatives, record:

- Source and access method
- Applicable license or terms
- Attribution and linking requirements
- Restricted or user-associated fields
- Refresh or deletion obligations
- Whether the offline edition republishes the material

Technical access is not permission to redistribute.

## Tooling and formatting

Application and deployment dependencies live in the root `package-lock.json`. Formatting and browser-test dependencies live in `tools/package-lock.json` and must use exact versions so local development and CI resolve the same toolchain without expanding the production dependency tree.

Use:

```bash
npm ci
npm run tools:install
```

The canonical formatter configuration is `prettier.config.mjs`. It loads `prettier-plugin-astro` and explicitly selects the Astro parser for `.astro` files. Generated wrappers, downloads, build output, and lockfiles may remain excluded through `.prettierignore` when formatting them would create noise or edit generated artifacts.

Before opening or merging a pull request that changes maintained source or documentation:

```bash
npm run format
npm run format:check
npm run check
```

Interactive labs also install the matching browser binary and run the smoke suite:

```bash
npm run --prefix tools playwright:install -- chromium
npm run test:smoke
```

Do not use floating `npx --yes` formatter commands or temporary `npm install --no-save` browser-test dependencies in normal validation workflows.

## Validation and release

Every pull request should pass:

- Astro-aware formatting checks
- Experiment generation
- JavaScript syntax validation
- Astro content validation and production build
- Playwright smoke tests for interactive labs

External-link checks run on a schedule rather than blocking ordinary pull requests.

## Lifecycle

Statuses:

- **Idea:** documented but not yet runnable
- **Prototype:** runnable but actively changing
- **Active:** intentionally maintained
- **Paused:** retained with no current work planned
- **Archived:** preserved as a historical artifact
- **Graduated:** moved to a permanent destination while the lab page remains as a stable pointer

Graduated experiments require `destinationUrl`. Archive or graduation should not silently break existing lab URLs.
