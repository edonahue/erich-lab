Architecture and build plan

Parse broadly, normalize carefully, derive graphs late, and publish a reliable static experience before depending on a live home service.

1. Public and private boundaries

Public pages explain logical roles, technologies, algorithms and measured outcomes. Exact inventory, addresses, storage paths, deployment commands, backup procedures and incident notes belong in a separate private infrastructure and data-operations repository.

PublicPrivate
Four Raspberry Pi 3B+ workers, SSD-backed x86 control node, optional workstation-class build node and compact multi-gig wired fabric.Router and switch models, port maps, hostnames, addresses, mount paths, credentials and physical cable routing.

2. Collection-first scale path

  1. Collection plus one hop: prove aliases, credits, evidence and a playable path.
  2. Representative medium graph: stress hubs, role rules, components and indexes.
  3. Full catalog: scale after schemas and snapshot contracts stabilize.

Storage is not the limiting factor; iteration speed and reproducible correctness are.

3. Data foundation

XML.gz → lxml iterparse → Parquet → DuckDB → graph snapshots

Broad normalized Parquet is the durable pivot layer. DuckDB handles analytical validation and transformations. PostgreSQL stores artist search, challenges, snapshot registry, accepted paths, caches and game state.

4. Graph design

The graph remains bipartite: artist → release → artist. NetworkX is used for small fixtures and correctness tests. Production snapshots use versioned integer adjacency arrays opened with NumPy memory maps and accompanied by checksums and provenance.

5. Runtime architecture

Control and storage

Docker Swarm manager, PostgreSQL, Redis/RQ, canonical snapshots, API and publisher.

Worker pool

Bounded independent batches on four Raspberry Pi 3B+ nodes.

Pi 1Pi 2Pi 3Pi 4

Heavy build node

Optional full ingest, compaction, image development and expensive analysis; not required for uptime.

Ansible provisions the machines. Docker Swarm places services. Redis and RQ distribute independent work. Dask remains an optional advanced comparison for workloads that genuinely need coordinated distributed analytics.

A compact multi-gigabit switch provides a stable local fabric between the control node and the existing wired worker-rack uplink. The public design records its role, not its consumer model or port map.

7. Product sequence

  1. Static MVP: daily or curated challenge, precomputed evidence paths and research findings.
  2. Committed phase two: arbitrary two-artist search through a bounded FastAPI service.
  3. Later pivots: manual relay, hidden contributor, role constraints and collection-specific play.

Album art is not a launch dependency; generated or typographic release tiles keep the interface independent of unresolved image rights.

8. Public delivery

Astro provides the site shell, with Svelte and TypeScript for the interactive game and accessible SVG for evidence paths. Cloudflare hosts static artifacts. A later Cloudflare Tunnel exposes only the bounded API, while Tailscale remains the private administration channel. Static challenges remain online when the home service is unavailable.

9. CI, testing and monitoring

Build

GitHub Actions, Docker Buildx and GHCR publish AMD64 and ARM64 images. Native Pi builds are a study exercise.

Quality

pytest, Playwright, Ruff and targeted mypy checks cover data, API and browser behavior.

Health

Structured logs, health endpoints, queue depth, snapshot version and a sanitized public status artifact.

11. Selected stack

Core: Python 3.12, Ansible, Docker Swarm, Redis/RQ, lxml, Parquet, DuckDB, PostgreSQL, NumPy memory maps and NetworkX fixtures.

Application: FastAPI, Pydantic, Astro, Svelte, TypeScript, SVG and selected D3 utilities.

Delivery: Cloudflare static hosting, later Cloudflare Tunnel, Tailscale, GitHub Actions, Buildx, GHCR, pytest and Playwright.

Optional study: Dask Distributed after the core queue works.
